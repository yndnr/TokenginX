# 开源协作规范

## 代码审查（Code Review）

- 所有代码必须通过 Pull Request 提交
- 至少需要一个维护者的审查批准
- 关注代码质量、性能影响、安全性
- 使用 GitHub Review 功能提供建设性反馈
- 审查重点：
  - 是否符合项目架构和设计模式
  - 是否有充分的单元测试
  - 是否有性能影响（必要时要求提供基准测试）
  - 是否引入安全风险
  - 代码可读性和可维护性

## Issue 管理

- 使用 Issue 模板报告 Bug 或提出功能请求
- Bug 报告应包含：复现步骤、预期行为、实际行为、环境信息
- 功能请求应包含：使用场景、解决的问题、预期实现方式
- 使用标签分类：`bug`、`enhancement`、`documentation`、`good first issue`

## 文档要求

- 公共 API 必须有 GoDoc 注释
- 复杂算法需要注释说明原理
- 新功能需要更新 readme.md 和相关文档
- 重大变更需要更新 changelog.md
- 所有 Markdown 文件使用小写文件名（如 readme.md、changelog.md、contributing.md）

## 许可证合规

- 确保依赖库的许可证兼容
- 不引入 GPL 等传染性许可证的依赖
- 所有源代码文件头部包含许可证声明（如需要）

## 社区行为准则

- 遵循 GitHub 开源社区准则
- 尊重贡献者，提供建设性反馈
- 欢迎新手贡献，标记 `good first issue`
- 及时响应 Issue 和 Pull Request
